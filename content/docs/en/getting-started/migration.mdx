---
title: Migration from NextAuth v4
description: How to migrate from NextAuth v4 to VinextAuth.
---

VinextAuth is designed as a drop-in replacement for NextAuth v4. Most apps require only two changes:

## 1. Replace the package

```bash
npm uninstall next-auth
npm install vinextauth
```

## 2. Update imports

| NextAuth v4 | VinextAuth |
|---|---|
| `import NextAuth from "next-auth"` | `import VinextAuth from "vinextauth"` |
| `import { getServerSession } from "next-auth"` | `import { getServerSession } from "vinextauth/server"` |
| `import { useSession } from "next-auth/react"` | `import { useSession } from "vinextauth/react"` |
| `import { SessionProvider } from "next-auth/react"` | `import { SessionProvider } from "vinextauth/react"` |
| `import GoogleProvider from "next-auth/providers/google"` | `import Google from "vinextauth/providers/google"` |

## 3. Update the API route

```ts title="Before (NextAuth v4)"
// src/pages/api/auth/[...nextauth].ts
import NextAuth from "next-auth";
import { authOptions } from "@/auth";

export default NextAuth(authOptions);
```

```ts title="After (VinextAuth)"
// src/pages/api/auth/[...vinextauth].ts
import { GET, POST } from "@/auth";

export { GET, POST };
```

## What stays the same

- All callback names and signatures (`signIn`, `session`, `jwt`, `redirect`)
- Session shape (`session.user.name`, `session.user.email`, `session.user.image`)
- Cookie format — existing sessions remain valid
- Provider configuration shape

## What's different

- No Node.js `crypto` — uses Web Crypto API
- No `pages/api/auth/[...nextauth]` — use `[...vinextauth]` to match Vinext routing
- Credentials provider requires CSRF token by default (matches NextAuth behavior)
- `getServerSession` requires the auth config, not `authOptions`

## Environment variables

VinextAuth uses its own env var names — **not** `NEXTAUTH_*`:

| NextAuth v4 | VinextAuth |
|---|---|
| `NEXTAUTH_SECRET` | `VINEXTAUTH_SECRET` |
| `NEXTAUTH_URL` | `VINEXTAUTH_URL` |

## Known limitations

- No database adapter for Prisma/Drizzle (only Cloudflare KV and D1)
- No email provider via Nodemailer (uses edge-compatible fetch-based transport)
- No `unstable_getServerSession` — use `getServerSession` directly
